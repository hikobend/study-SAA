## SAA#20(Auto Scaling)

・Auto Scalingの設置数

Webサーバーとアプリケーションサーバーに設置する。データベースには設置する必要なない。

・Auto Scalingグループに適切なEC2インスタンスがあるようにする

ClouWatchアラームの作成。EC2インスタンスをアプリケーション層へ参加。EC2インスタンスのCPU使用率メトリクスの使用。Auto Scalingで別のインスタンスの起動。

・Auto ScalingでEC2インスタンスを軌道から実行までにかかる時間

基本的に数分。

・Auto Scalingのクールダウン期間の設定

・デフォルトのクールダウン

・RAND　0

・S3のデータ損失から保護する

## SAA#116
 
・ミッションクリティカル

業務の遂行やサービスに必要不可欠であり、障害や誤作動などが許されない。

・署名つきURL

ユーザーはウェブサーバー経由でリクエストをルーティングせず、S3に直接アップロードできる。

・スポットインスタンスの中断操作

中断中にEC2がスポットインスタンスを休止、停止、終了するかを指定できる。中断時の動作を選択できる。中断したスポオットインスタンスを停止させるには、タイプをpresistentにする

・CloudTrail

KMSを用いてで管理されたキー(SSE-KMS)によるCloudTrailログファイルを暗号化することで、データセキュリティの要件を満たせる。

・サーバー側の暗号化(SSE-C)

お客様が用意したキーによるサーバー側の暗号化を使用する場合に有効。

・クライアント側の暗号化(CSE-C)

クライアント側の暗号化では、S3に送信する前にデータを暗号化する。

・APIGateway

基盤となる複数のバックエンドサービスに単一のインターフェースを提供することにより、ユーザーインターフェースの機能をその基盤サービスから切り離せる。

## SAA#117

・S3リソースにアクセス許可を付与

バケットポリシーとユーザーポリシーはS3リソースにアクセス許可を付与するために使用。

・別のリージョンにスナップショットを作成する

Redshiftクラスターのスナップショットを自動的に別のAWSリージョンにコピーするようにRedshiftを設定でき、Redshiftクラスターはコピーされたリージョンのスナップショットから復元できる。

・SQS

Lambda関数を作成し、SQSを導入することで、データベースに問題を発生した場合でも、再試行で顧客データを保存することができる。

## SAA#118

・暗号化を使用したデータ保護

送信する前にクライアント側で暗号化し、クライアント側の暗号化を用いることでS3に送信する前にデータを暗号化できる。

・デフォルトの終了ポリシー

AZ間のインんスタンス数が異なる場合、インスタンスが最も多く、スケイルインから保護されていないインスタンスが1つ異常あるAZを決定して終了する。

・ECSの特徴

ECSのFargate起動タイプを使用すると、メンテナンスとスケーリングのための継続的な努力を最小限に抑えたサーバーレスクラスターを提供。

## SAA#119

・ディサイダー

・Athena

・VPCエンドポイント
